# DomStudio Social Media / Influencer Affiliate Roadmap

Draft v1 — 2026-07-01

## 0. Premise

One coordinator recruits and manages a roster of 10–30 people who have a
social media following relevant to DomStudio's audience (WB/Ozon/Yandex/Avito
sellers, plus the beauty/food/auto small-business verticals already built
into the copy-tool suite). Each person in the roster gets a personal discount
code. Customers who use that code get a discount; the person who owns the
code earns a commission; the coordinator earns an override on top of the
whole group's activity.

Commission structure (confirmed):

- **Coordinator: 15% override** — on all sales generated by every code in
  her group, paid on top, does not reduce anyone else's cut.
- **Influencer: 15% direct** — on sales generated by their own personal
  code.
- **Customer: 15% off first purchase** — locked for Phase 1 so the pilot
  has clear unit economics before codes go live.

## 1. What already exists vs. what needs to be built

DomStudio already has a referral system (commit `67c34e5`, live) that is
**structurally close** to what this program needs but solves a different
problem:

- `referral_code` — an 8-char code every user already has (`database.py`,
  `routers/auth.py`)
- `referred_by_code` — set at registration if someone signed up through a
  link
- Reward: flat **+500 tokens** to both sides on first verification — no
  percentage discount, no percentage commission, no two-level hierarchy
  (coordinator → influencer → customer), no cash payout ledger

That system is for casual user-to-user invites. It cannot be reused as-is
for this program because it has no concept of a discount percentage, a
commission percentage, or a parent/child code relationship. But the schema
pattern (a code table + attribution on signup/checkout) is the right shape
to extend rather than replace.

**New backend work required:**

1. `promo_code` table: `code`, `owner_label` (influencer name/handle —
   these people will often not be registered DomStudio users), `parent_code`
   (nullable — set for influencer codes, points at the coordinator's master
   code), `discount_percent`, `commission_percent`, `active`
2. Checkout flow (`routers/payments.py`) needs to accept an optional promo
   code, apply the discount to `price_rub` before charging Tinkoff, and
   record which code was used against the resulting payment
3. Commission ledger: `payment_id`, `code`, `base_amount_rub`,
   `commission_rate`, `commission_amount_rub`, `payee`, `payout_status` —
   two rows get written per qualifying sale (the influencer's 15% row, and
   the coordinator's 15% override row on the same sale)
4. Reporting: a plain internal view (Supabase table browser is enough at
   this volume — 10-30 codes) rather than a customer-facing dashboard. Build
   a real dashboard only if/when this scales past manual tracking.

This is new backend work, not a copy-paste of the existing referral system.
Budget it as its own small feature rather than assuming it rides for free
on top of `67c34e5`.

## 2. Roles

### Coordinator

- Recruits and manages the roster (target 10–30 people)
- Gets one master code that every influencer's code is a child of
- Earns 15% override on every sale attributed to any code in her group
- Owns onboarding: hands each influencer their code, a short pitch deck,
  and sample post copy (see §6)
- Single point of contact for reporting questions — she should not need
  backend access; a monthly summary is enough

### Influencers (10–30 people)

- Each gets one unique code + shareable link
- Posts about DomStudio's actual, live capabilities — see §3 for what's
  safe to claim
- Customer using their code gets the agreed discount on first purchase;
  influencer earns 15% of that purchase
- Recommend capping the commission window to **first purchase only** (not
  lifetime recurring), to keep payout liability predictable while the
  program is unproven

## 3. What influencers are allowed to claim (honesty guardrail)

This pulls directly from the standing project rule: **be honest, don't
oversell a mode past what it actually does.** Concretely, for this
audience:

| Claim | Status | Notes |
|---|---|---|
| "AI cleans up your product photos for marketplace listings" | Safe | Catalog mode — live, verified |
| "AI puts your product in a real scene / lifestyle shot" | Safe | Product/lifestyle img2img — live, product identity preserved |
| "AI writes your Avito/Ozon/WB/Yandex Maps listing copy" | Safe | 14 copy tools across beauty/food/auto verticals — live |
| "AI generates video for your listing" | **Careful** | WanVideo pipeline works on the Turing GPU box but is not the default always-on path — confirm current status before an influencer promises "video" in a post; do not let anyone imply a still image with pan/zoom is a video |
| "Virtual try-on" | **Careful** | Only claim this if the fitting workflow being demoed is the real virtual try-on model, not generic image edit |

Give each influencer 2-3 short example posts using real before/after
assets already in `temp-preview/` / the curated gallery, not stock claims.

## 4. Unit economics and pilot rules

These rules are locked for Phase 1 so the program can launch without fuzzy
money math:

- **Customer discount:** start at **15%** for the pilot, not 20%.
- **Commission base:** calculate influencer and coordinator commission from
  the **actual paid amount after discount**, not the original list price.
- **Commission window:** commission applies to the customer's **first paid
  purchase only**, not lifetime recurring spend.
- **Stacking:** one promo code per customer purchase; promo codes cannot stack
  with other discounts or referral bonuses.
- **Self-referral:** influencers, the coordinator, and internal test accounts
  cannot earn commission on their own purchases.
- **Controls:** every code needs `active`, `starts_at`, and optional
  `expires_at` / `paused_reason` fields so a code can be shut off quickly.

The pilot should be judged as paid acquisition spend. With a 15% customer
discount, 15% influencer commission, and 15% coordinator override, the maximum
first-purchase acquisition cost is 45% of the post-discount paid amount before
payment fees and AI generation costs. That is acceptable only because the
commission window is first-purchase-only and the first pilot is capped at 3-5
influencers.

## 5. Discount code mechanics

- Customer applies code at checkout → gets 15% off first purchase
  (**15% for Phase 1**) before Tinkoff payment is created
- Code is tied to one influencer; influencer's code is tied to the
  coordinator's master code for the override
- First purchase only, per customer; prevent stacking, reuse, and self-referral
- Track: signups via code, first purchase via code, discount given,
  commission owed (influencer + coordinator rows)
- Track funnel performance per code: signup, checkout started, paid, refunded,
  commission pending, commission approved, commission paid

## 6. Recruitment plan for the coordinator

- She sources candidates from her own network first — target people whose
  audience already includes WB/Ozon/Avito sellers, or beauty/food/auto
  small-business owners (this maps directly onto the 3 verticals already
  live in the copy tools)
- Give her a one-page pitch covering: what DomStudio does, the discount the
  customer gets, the 15% the influencer earns, and 2-3 real before/after
  examples
- Each influencer, once recruited, gets: their code + link, the honesty
  guardrail table above, and 2-3 sample captions in Russian

## 7. Content cadence template (per influencer, repeatable monthly)

| Week | Post type |
|---|---|
| 1 | Before/after catalog cleanup (real product photo) |
| 2 | Lifestyle/product-in-scene shot |
| 3 | Copy-tool demo — screenshot of an AI-written listing for their niche (beauty/food/auto) |
| 4 | Discount code reminder / short testimonial |

Vertical short-form (Stories/Reels format) can slot into any week once the
video pipeline status is confirmed current — don't schedule a video post
in the template until that's verified live, per the honesty guardrail.

## 8. Payout logistics

- Calculate monthly, from the ledger in §1, using the post-discount paid amount
  as the commission base
- Pay influencers directly for their 15%; pay the coordinator her 15%
  override as a separate line, not netted through the influencers
- Set a minimum payout threshold (e.g. skip payout below some small ruble
  amount, roll it to next month) to avoid micro-transfers
- Flag for the user: paying individuals in Russia for this kind of
  commission work generally implies self-employed (самозанятый) status on
  their side — worth confirming how payouts will be documented before
  scaling past the pilot, this is an administrative/tax question for the
  user to resolve, not something to route around

## 9. Rollout phases

**Phase 0 (now):** Structure agreed (this doc). Coordinator briefed.

**Phase 1 (pilot, ~2 weeks):** Build the promo-code + ledger backend
(§1 and §4). Issue codes to a small pilot group (3-5 influencers) before the
full 10-30, so the discount, commission math, anti-abuse rules, and checkout
integration get a real test with low exposure.

**Phase 2 (full rollout):** Once pilot numbers look sane, onboard the rest
of the 10-30 roster. Content calendar (§7) starts in earnest.

**Phase 3 (month 2+):** Review conversion rate and margin impact per code.
Adjust discount % or commission % if the unit economics are off. Consider
a lightweight self-serve reporting view for the coordinator only if manual
tracking becomes the bottleneck.

## Phase 1 build order

1. Add `promo_code` storage with parent-code support and active/pause fields.
2. Add checkout promo validation and 15% discount calculation.
3. Store the promo code and paid amount on successful payment.
4. Write two commission-ledger rows after successful payment: influencer 15%
   and coordinator 15%.
5. Add a simple internal report/export for signups, paid purchases, discounts,
   and payout status.
